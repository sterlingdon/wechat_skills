# 📝 WeChat Official Account Manager (微信公众号全自动运营管家)

## 1. Skill 概述

本 Skill 专为 AI Agent 设计，将繁琐的微信公众号运营抽象为一条极其流畅的自动化流水线。

**核心能力：**

- 🧠 **AI 自主风格分析** — 自动分析文章情感意境，选择匹配的排版样式，生成封面建议和摘要
- 🎨 **Gemini 官方 SDK 绘图** — 底层深度整合 `google-genai` SDK，生成微信头条封面（900x383）及正文插图（1080x608）
- 🪄 **微信专属黑魔法排版** — 自动将 Markdown 渲染为 HTML，并通过 `premailer` 将 CSS 强制注入为内联样式
- 🔄 **图床置换与无缝入库** — 自动上传图片至微信素材库，调用草稿 API 一键推送

## 2. 目录结构

```
content_hub/
  00-account-profile/                # 账号主设定 / 人设 / 栏目策略 ⭐
    ai-result-bro.md                 # 当前账号人设（如仅 1 份，则默认激活）
  01-ideas/                          # 灵感记录
    YYYY-MM-DD/                      # 按日期组织

        idea.md                      # 灵感主文件（下游主入口）

        01.jpg, 02.jpg...            # 附件图片（如有）
        
  02-drafts/                         # 初稿
    YYYY-MM-DD/
      {article-id}/                  # 必须沿用 ideas 阶段同名目录 ⭐
        {article-id}.md              # Markdown 文章

  03-review/                         # 待审核（预览阶段）

      {article-id}/
        {article-id}.md              # Markdown 文章
        {article-id}_preview.html    # 预览 HTML
        images/                      # 配图目录
        sync_record.json             # 同步记录（如有）
        
  04-published/                      # 已发布归档
    YYYY-MM-DD/
```

> ⚠️ **重要规则 1：`01-ideas` 必须按“每天一个日期目录 + 每题一个文章目录”的结构组织。** 每篇文章使用独立目录，以 `{article-id}` 命名。这个目录名从 ideas 阶段开始就是全流程唯一 ID，后续进入 drafts、review、published 时 **不得重命名**。
>
> ⚠️ **重要规则 2：如果 `content_hub/00-account-profile/` 下只有 1 份 `.md` 文件，则默认将其视为当前账号的激活人设文件。写稿前必须先读取它，再读取 `idea.md`。**

## 4. 执行流程

### 【第一步】内容创作

> ⚠️ **创作前置要求：必须先读取仓库根目录 `content_hub/01-ideas/` 下对应的灵感文件，以其为大方向锚点，再主动联网搜索更多资料，综合后进行写作。禁止直接复制 ideas 内容，也禁止完全脱离 ideas 方向凭空创作。**

**智能体操作步骤：**

1. **先读取账号主设定（人格锚点）**
   - 检查 `content_hub/00-account-profile/`
   - 若其中只有 1 份 `.md`，默认将其视为当前账号的激活人设文件，必须先读取
   - 读取时重点识别：
     - **账号一句话定位**：这号到底是干什么的？
     - **角色气质**：像专家、朋友、同事，还是媒体？
     - **口吻约束**：哪些表达鼓励用，哪些表达要避免
     - **内容层级 / 栏目**：这篇更该归入口层、样板层还是扩展层
     - **选题 / 标题 / 结构铁律**：什么样的文章更像这个号
   - 如果没有账号主设定文件，则退回使用本 skill 内置规范

2. **定位灵感文件（方向锚点）**
   - 优先查找 `content_hub/01-ideas/YYYY-MM-DD/` 下与当前任务匹配的 `{article-id}/idea.md`
   - `daily-overview.md` 只用于浏览和挑题，不作为最终写作锚点
   - 若用户未指定日期，先读取对应日期下的 `daily-overview.md`
   - 默认向用户展示当日 Top 3 候选题；若用户说“你帮我挑”，优先选择排名最高且传播潜力最强的一题
   - 若 `01-ideas` 目录为空或无对应文件，**停止流程并告知用户**，请用户先补充灵感素材

3. **解读灵感意图（不是抄内容，是理解方向）**
   - 读取 `idea.md`，识别：
     - **核心话题**：文章要聊什么？
     - **写作立场/观点倾向**：作者的态度是什么？
     - **目标读者**：写给谁看？
     - **关键词 / 关注焦点**：哪些词、事件、概念是核心？
     - **人设适配理由**：为什么这个题适合当前账号写？
     - **建议内容层级**：入口层 / 样板层 / 扩展层
     - **建议栏目归属**：更适合放进哪个固定栏目？
     - **推荐结构**：建议按什么段落框架展开？
     - **开头切入建议**：最适合从哪个画面、冲突或判断切入？
   - `idea.md` 是初始方向锚点，**不要照搬**，用它来校准后续搜索和写作的方向
   - 如需补足来源、时间线或引用，可再查看同目录下的 `sources.md`
   - 如果联网补充后发现某些事实需要修正，可以修正事实和结论；但**不能把题写偏成另一个主题**

   > ⚠️ **资源理解要求：ideas 文件中如有引用任何附件（图片、截图、链接等），必须主动读取并理解，不能只看文字。**
   >
   > - **图片 / 截图**（如 `./01.jpg`、`![](image.png)` 等）：使用视觉理解能力逐张查看，提取图片中展示的界面、内容、数据等关键信息，这些往往是用户最直接的证据和素材
   > - **本地链接 / URL**：尝试访问并提取内容
   > - **理解深度要求**：图片所传达的信息与文字描述同等重要，必须结合图片内容来理解用户意图，不得忽略

4. **联网搜索，补充真实资料**
   - 以灵感文件的核心话题和关键词为线索，主动搜索互联网
   - 搜索目标包括但不限于：
     - 最新新闻、事件、数据
     - 权威报告、研究结论
     - 相关案例、真实故事
     - 不同角度的观点和争议
   - 筛选与 ideas 方向一致的内容纳入素材库，过滤无关信息

5. **综合撰写正文**
   - 在 `content_hub/02-drafts/YYYY-MM-DD/` 下创建对应 Markdown 文章
   - 以账号主设定的**人格与价值交付方式**为外壳，以 ideas 的**大方向和写作意图**为骨架，以**联网搜索获取的资料**为血肉，综合写作
   - 观点有据可查，数据有来源，内容有深度，不能只复述 ideas 原文
   - 可对素材进行提炼、组织、结构化和润色
   - 如果账号主设定与 `idea.md` 出现轻微冲突，优先保证文章仍然像当前账号在说话；如果冲突严重，应先停下来重新挑题，而不是硬写

   > ⚠️ **Markdown 格式规范（必须遵循）**：
   > 
   > 文章必须有清晰的结构层次和视觉重点，便于读者快速抓住核心信息。以下是强制要求：
   >
   > **标题层级**：
   > - `#` 主标题（仅一个，文章标题）
   > - `##` 一级小标题（核心论点/章节划分）
   > - `###` 二级小标题（细分要点，可选）
   > - 每个标题下应有完整段落，不要出现"空标题"
   >
   > **重点强调**：
   > - **核心观点**、**关键数据**、**重要结论** 必须用 `**加粗**` 标注
   > - 引用内容、金句可用 `> 引用块` 格式突出
   > - 不要滥用加粗，每段最多 1-2 处重点
   >
   > **结构化表达**：
   > - 并列观点用**无序列表**（`-` 或 `*`）
   > - 步骤流程用**有序列表**（`1.` `2.` `3.`）
   > - 对比内容用**表格**或并列结构
   > - 复杂概念可分点阐述，每点 1-2 句
   >
   > **示例**：
   > ```markdown
   > # 深入理解 React Hooks
   > 
   > React Hooks 改变了我们编写组件的方式...
   > 
   > ## 核心概念
   > 
   > Hooks 让函数组件拥有了状态管理能力：
   > - **useState** — 管理组件状态
   > - **useEffect** — 处理副作用
   > - **useContext** — 跨组件共享状态
   > 
   > > Hooks 本质是把组件逻辑"挂载"到函数上。
   > ```

   > ⚠️ **文章结尾公共模板（必须添加）**：
   > 
   > 每篇文章结尾必须包含引导关注的内容，与文章主体无关。以下提供多个版本，**任选其一**：
   > 
   > **版本 1（直接型）**：
   > ```markdown
   > ---
   > 
   > **如果觉得这篇文章对你有帮助，欢迎点赞、在看、转发三连支持一下～想第一时间看到后续更新，记得星标公众号哦。感谢阅读，我们下篇见。**
   > ```
   > 
   > **版本 2（温和型）**：
   > ```markdown
   > ---
   > 
   > **看到这里了，觉得有收获的话，不妨点个赞或在看。关注我，第一时间获取更多干货内容。感谢你的时间。**
   > ```
   > 
   > **版本 3（互动型）**：
   > ```markdown
   > ---
   > 
   > **如果你也觉得这个话题有意思，评论区聊聊？点赞、在看、转发任选一项都是对创作者最大的鼓励。点个星标，下次更新不迷路。**
   > ```
   > 
   > 这个结尾用于：
   > - 引导用户互动（点赞、在看、转发、评论）
   > - 引导持续关注（星标公众号）
   > - 增强用户粘性和文章传播力
   > 
   > **注意**：可根据文章风格选择合适版本，核心意思保持一致即可。

   > ⚠️ **Markdown 格式规范（必须遵循）**：
   > 
   > 文章必须有清晰的结构层次和视觉重点，便于读者快速抓住核心信息。以下是强制要求：
   >
   > **标题层级**：
   > - `#` 主标题（仅一个，文章标题）
   > - `##` 一级小标题（核心论点/章节划分）
   > - `###` 二级小标题（细分要点，可选）
   > - 每个标题下应有完整段落，不要出现"空标题"
   >
   > **重点强调**：
   > - **核心观点**、**关键数据**、**重要结论** 必须用 `**加粗**` 标注
   > - 引用内容、金句可用 `> 引用块` 格式突出
   > - 不要滥用加粗，每段最多 1-2 处重点
   >

   > - 并列观点用**无序列表**（`-` 或 `*`）
   > - 步骤流程用**有序列表**（`1.` `2.` `3.`）
   > - 对比内容用**表格**或并列结构
   > - 复杂概念可分点阐述，每点 1-2 句
   >
   > **示例**：
   > ```markdown
   > # 阶跃AI的龙虾太猛了，stepclaw正在扒光我的底裤
   > 
   > 今天体验了阶跃AI的全局记忆功能，我直接傻眼了。
   > 
   > ## 惊喜：AI真的懂你了
   > 
   > 这个"龙虾"会定期给我的屏幕截图，自动分析我在干嘛：
   > - **活动记录**：我在哪编辑了代码、开了什么终端
   > - **时间线生成**：自动整理一天的工作轨迹
   > - **视频回顾**：甚至还能生成回顾视频

5. **处理配图（全文写完后统一处理，分五个优先级）**

    > ⚠️ **配图占位符必须在全文完成后，通读全文、理解每个段落主题后，再逐段决策插入。不能在写作过程中随意嵌入。**

    **优先级 0：复用 ideas 目录中的证据图 / 原图（最高优先级）⭐**
    - 在撰写配图占位符前，**必须先检查 ideas 目录中是否有可用的原始图片或证据截图**
    - 使用**视觉能力**逐张查看 ideas 目录中的图片（`01.jpg`、`02.jpg` 等）
    - 判断标准：
      - 图片内容是否与当前段落主题相关？
      - 图片质量是否足够清晰？
      - 图片是否包含敏感信息（密码、隐私数据等需要打码）？
    - 如果图片可用：
      - 直接复制到 `content_hub/02-drafts/YYYY-MM-DD/{article-id}/images/` 目录
      - 在文章对应位置用标准 Markdown 图片语法引用：`![说明文字](images/filename.jpg)`
      - **不要为了用图而用图**，图片必须与段落内容相关

    **优先级 1：检查 drafts 目录中已有的图片（爬虫产出）⭐**
    - 检查 `content_hub/02-drafts/YYYY-MM-DD/{article-id}/images/` 目录，是否已有图片
    - 这些图片可能来自：
      - 之前联网搜索时爬虫下载的图片
      - 用户手动放入的素材图片
    - 使用**视觉能力**逐张查看已有图片，判断是否可用于当前段落
    - 判断标准：
      - 图片内容是否与段落主题相关？
      - 图片质量是否足够清晰？
      - 是否需要裁剪/打码处理？
    - 如果图片可用，直接在文章对应位置引用，无需重复生成

    **优先级 2：下载联网搜索资料中的原图**
    - 在第3步联网搜索时，若引用了某篇文章或报告，**竭尽全力尝试下载其中的图片**
    - 下载成功的图片，直接保存到 `content_hub/02-drafts/YYYY-MM-DD/{article-id}/images/` 目录
    - 在文章对应位置用标准 Markdown 图片语法引用：`![说明文字](images/filename.jpg)`

    **优先级 3：写 AI 生成 prompt（原图不可得时）**
    - 若某处确实需要配图，但原图无法获得，则在对应段落后插入 prompt 占位符
    - **prompt 必须基于对该段落的完整理解来撰写**，而非模板套用
    - 科技文章的 prompt 要求：**PPT 信息图风格**，图片内容要能清晰展示对应段落的核心逻辑，不得生成与段落无关的场景插画
    - **图片语言规则**：
      - 文章主体是**中文** → 图片中的文字、标签、说明必须使用**中文**
      - 文章主体是**英文** → 图片中的文字、标签、说明必须使用**英文**
    - 占位符格式：
      ```markdown
      <!-- image: [具体描述段落核心信息的 infographic prompt] -->
      ```

    **优先级 4：不插图（无必要时）**
    - 若某段内容不适合配图（如纯抒情、过渡段、结尾），**不要强行插图**

    **配图工作流（智能体必须遵循）**：
    ```
    1. 检查 ideas 目录 → 有图片？→ 用视觉能力评估 → 可用？→ 复制使用
                                              ↓ 不可用/无图
    2. 检查 drafts/images 目录 → 有图片？→ 用视觉能力评估 → 可用？→ 直接引用
                                              ↓ 不可用/无图
    3. 检查联网搜索资料 → 有图片？→ 尝试下载 → 成功？→ 保存使用
                                              ↓ 失败
    4. 判断是否需要配图 → 需要？→ 写 AI prompt（语言匹配文章语言）
    5. 不需要 → 不插图
    ```

    > ⚠️ **核心原则：能用原图就不用 AI 生成**。原图更真实、更具体、更有说服力。

    **配图占位符格式（优先级 2 使用）：**
    ```markdown
    <!-- image: 画面详细描述 -->
    ```

    **示例（中文文章）：**

## 5. 科技文章写作风格规范

> ⚠️ **适用范围**：本规范仅适用于**科技、AI、编程、工具推荐**类文章。生活/情感类文章不在此范围内。
>
> ⚠️ **优先级说明**：如果存在 `content_hub/00-account-profile/*.md` 账号主设定文件，则账号主设定优先于本章默认风格规范。本章只作为兜底风格，而不是覆盖账号人格。

### 5.0 风格核心：先像账号本人，再像默认科技作者

**基调定位**：
- **亲切不装**：像技术朋友聊天，不端着，不说教
- **专业通俗**：有技术深度，但表达接地气，小白也能看懂
- **真诚坦率**：不回避问题，优缺点都说，不无脑吹捧

### 5.1 标题写法

**公式**：`动词/数字` + `核心价值` + `制造期待`

**好的标题**：
```
✅ 安利一个11万Star的必装插件，能让你的Agent体验直接质变
✅ 我用Claude Code开发了一个ADHD阅读器，踩坑踩到汗流浃背
✅ Superpowers到底香不香？我替你们试了一个月
```

**差的标题**：
```
❌ Superpowers插件介绍（太平淡）
❌ 如何使用Claude Code开发应用（太泛）
❌ AI Agent开发最佳实践（太官方）
```

### 5.2 开头写法

**三步公式**：
1. **抛出痛点/背景**：让读者产生共鸣
2. **引出主题**：告诉读者你要聊什么
3. **制造悬念**：让读者想往下看

**示例**：
```markdown
最近一直在聊Agent、聊Vibe Coding。

但是在给越来越多的朋友安利的时候，发现其实，一直有一个问题被忽略了。

就是，真正卡住大多数人的，是自己没有一个标准的工作流程。

所以，我想给大家分享一个我自己一直在用的超好用的插件...
```

### 5.3 正文写法

**段落结构**：
- 每段 3-5 句，不要太长
- 用 **加粗** 强调核心观点（每段最多 1-2 处）
- 适当使用口语化表达

**口语化表达（推荐使用）**：
```
✅ "坦诚的讲..."
✅ "我觉得..."
✅ "说实话..."
✅ "就..."
✅ "其实是..."
✅ "我当时就傻眼了"
✅ "汗流浃背"
✅ "给我笑乐了"
```

**技术描述原则**：
- 先说"是什么"，再说"为什么重要"
- 用类比帮助理解（但不要过度类比）
- 具体的例子胜过抽象的描述

**示例**：
```markdown
**Superpowers其实不太能算一个传统意义上的工具，我觉得他更应该被定义为一套指导Agent如何完成任务的系统。**

因为坦诚的讲，绝大数的Agent，在进行任务的时候，天然都倾向于拿到任务就开始写代码，会跳过设计、跳过测试、跳过 review，然后产出一坨不可维护的东西。
```

### 5.4 结构化表达

**对比分析**：用表格或并列结构
```markdown
| 方案 | 优点 | 缺点 |
|------|------|------|
| 方案A | ... | ... |
| 方案B | ... | ... |
```

**步骤流程**：用有序列表
```markdown
1. 首先安装插件...
2. 然后重启...
3. 最后验证...
```

**功能介绍**：用无序列表
```markdown
这个工具有三个核心功能：
- **自动截图**：每隔N秒截图一次
- **时间线生成**：自动整理活动记录
- **视频回顾**：生成回顾视频
```

### 5.5 语气与态度

**推荐**：
```
✅ 展示踩坑经历，不装完美
✅ 承认自己也不懂某些地方
✅ 说"我觉得"而不是"你应该"
✅ 适度幽默，但不要刻意
```

**避免**：
```
❌ 官腔官调（"本文将介绍..."）
❌ 过度吹捧（"神器！秒杀一切！"）
❌ 也不说缺点（不真诚）
❌ 太多感叹号（显得不专业）
```

### 5.6 结尾写法

**三步公式**：
1. **总结核心观点**：用 1-2 句话概括
2. **给出建议/判断**：帮助读者做决策
3. **引导关注**：使用公共结尾模板

**示例**：
```markdown
所以现在我的感受是，AI来开发已经够快了，真正该花时间的地方是动手之前。

这个插件，是我推荐大家的，必装插件。

---

**看到这里了，觉得有收获的话，不妨点个赞或在看。关注我，第一时间获取更多干货内容。感谢你的时间。**
```

## 6. 图片风格一致性原则

> ⚠️ **严禁在同一篇文章中混用风格**（如：第一张极简冷蓝科技图，第二张温暖插画风，第三张赛博朋克）。读者视觉体验必须连贯统一。

## 7. 科技文章结构化配图规范

**核心原则：科技文章配图 = 信息可视化，而非场景插画。**

> ⚠️ **强制要求：科技文章的配图 prompt，必须在全文写完之后，通读整篇文章并分析每个段落的核心信息，再逐段判断是否需要配图、配什么类型的图。严禁在写作过程中随意堆砌 prompt。**
>
> **配图的本质是"把这段文字变成一张 PPT 幻灯片"：**
> - 每张图必须清晰传达对应段落的核心观点或逻辑结构
> - 读者看图不需要看文字也能理解这一段在讲什么
> - 图与段落之间不能有信息断层，不能只是氛围图或装饰图

### 7.1 配图类型分类

| 类型 | 适用场景 | 特点 |
|------|----------|------|
| **概念图** | 解释核心概念、原理 | 清晰的视觉隐喻、简洁图形、标注说明 |
| **流程图** | 步骤、流程、架构 | 箭头连接、模块化、层次分明 |
| **对比图** | A vs B、前后对比 | 左右/上下分割、清晰对比 |
| **架构图** | 系统架构、技术栈 | 分层结构、组件连接、技术栈展示 |
| **数据图** | 数据展示、统计信息 | 图表、数字、进度条、可视化 |

### 7.2 科技配图 Prompt 模板

> ⚠️ **图片语言规则（必须遵循）**：
> - 文章主体是**中文** → 图片中的文字、标签、说明必须使用**中文**
> - 文章主体是**英文** → 图片中的文字、标签、说明必须使用**英文**
> 
> 在 prompt 中必须明确指定语言，例如："中文标注" 或 "English labels"

**概念图（中文文章）：**
```
<!-- image: infographic 风格，中文标注。[概念名称]的可视化解释。中央是[核心元素]，周围标注"[关键要点1]"/"[关键要点2]"/"[关键要点3]"。简洁的图标、清晰的中文文字标签、冷色调蓝灰配色、白底、现代扁平化设计 -->
```

**概念图（英文文章）：**
```
<!-- image: infographic style, English labels. Visual explanation of [concept name]. Central [core element] surrounded by "[key point 1]"/"[key point 2]"/"[key point 3]". Clean icons, clear English text labels, cool blue-gray palette, white background, modern flat design -->
```

**流程图（中文文章）：**
```
<!-- image: infographic 风格，中文标注。[流程名称]流程图。从左到右展示"[步骤1]"→"[步骤2]"→"[步骤3]"，每个步骤用圆角矩形框表示，箭头连接，标注关键说明。简洁现代、蓝白配色 -->
```

**架构图（中文文章）：**
```
<!-- image: infographic 风格，中文标注。[系统名称]架构图。分层展示：顶层是"[表现层]"，中层是"[业务层]"，底层是"[数据层]"，用方框和连接线表示组件关系。技术蓝图风格、深蓝灰配色 -->
```

**对比图：**
```
<!-- image: infographic 风格。[A]与[B]对比图。左右分割布局，左侧列出[A的特点]，右侧列出[B的特点]，中间用VS分隔。清晰对比、现代简约 -->
```

### 7.3 配图 Prompt 要求

1. **必须包含 `infographic 风格`** — 触发信息图生成
2. **明确视觉结构** — 左右/上下/中心辐射/分层
3. **指定配色** — 必须与全文锁定的视觉基调一致（见第6节），冷色调/蓝灰/科技蓝等，全篇统一
4. **禁止模糊描述** — 不要"一个程序员坐在..."这种场景描述
5. **全篇视觉语言统一** — 同一篇文章所有 prompt 中的配色、字体风格、图形语言描述必须保持一致，写 prompt 时主动复用前一张图的风格描述词
6. **语言匹配文章语言** — 中文文章用"中文标注"，英文文章用"English labels"，图片中的所有文字必须与文章语言一致

### 7.4 智能体配图工作流

```
文章结构分析
     │
     ▼
┌─────────────────────────────────────┐
│  识别文章中的知识节点：              │
│  • 核心概念 → 概念图                │
│  • 步骤流程 → 流程图                │
│  • 对比分析 → 对比图                │
│  • 架构设计 → 架构图                │
│  • 数据统计 → 数据图                │
└─────────────────────────────────────┘
     │
     ▼
生成结构化 Prompt（infographic 风格）
```

**AI 智能体在 `01_article_analyze` 模块中分析文章类型，并在生成图片 prompt 时自动拼接对应的风格后缀，确保图片与文章气质一致。**
